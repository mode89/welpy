## Architecture

- Reference implementations consulted for behavior parity: dwl, sway, qtile. Source via nix: `nix-build '<nixpkgs>' -A dwl.src` (sway: `-A sway-unwrapped.src`; qtile: `-A python3Packages.qtile.src`).
- Pointer focus pipeline: `forward_pointer_motion(server, time)` re-points the seat pointer to the surface under the cursor; called from the pointer-motion handlers (`cursor_motion`, `cursor_motion_absolute`, and the `cursor_button` release path) and from `apply_focus` (re-sample after the scene changes with no mouse move). `surface_at` resolves the surface via `wlr_scene_node_at`.
- Scene hit-testing (`wlr_scene_node_at`) uses the committed surface size clamped by the clip; welpy always sets a clip (`wlr_scene_subsurface_tree_set_clip`), and the clip only *shrinks* the hit region, never grows it. So a window's hittable area lags its configured size until the client commits the new buffer.
- `focus_client` only bumps `client.focus_order`; all focus effects (keyboard focus, raise, border colors, and the pointer re-point) are emitted by `apply_focus` at the handler boundary.
- Outside an interactive grab (during a move/resize grab, motion drives `drag_client` instead), welpy re-points pointer focus on every motion â€” no implicit-grab / dwl-style "CurPressed" focus lock â€” so pointer focus is not held on a surface across a button press.
- `XdgClient.pending_serial` tracks a compositor-driven configure until the client renders it (the per-client analog of a transaction). Serial tracking and the screen-hold (`client_holds_paint`) are xdg-only; `X11Client` has no configure-serial tracking.
- Window tiling is a per-workspace n-ary tree: `Workspace.root` is a `layout.Container`, and `apply_geometry` walks it over `window_area` (equal splits; HORIZONTAL/VERTICAL containers) -- it replaced the old stateless `master_stack`. Pure tree logic lives in `layout.py` (leaves are opaque; it never imports `Client`); `wel.py` is the side-effecting boundary. `server.clients` is now just a registry; `focus_order` (a monotonic per-focus counter) drives focus/z-order and focus-on-close. The retired `master_stack`/`zoom`/`cycle_focus` are gone; tree keys are `mod+hjkl` directional focus, `mod+shift+hjkl` directional move (`layout.move`), `mod+v` wrap the focused window into a new group split along its long side, and `mod+e` flip a container's axis. There is no explicit ungroup -- groups dissolve only via the localized collapse in `remove`/`move`.
- `workspace.fullscreen` and per-window `floating_geom` are orthogonal: `client_layer` resolves a window as fullscreen â†’ float â†’ tile, and fullscreen overlays geometry without clearing `floating_geom`, so leaving fullscreen restores the prior float-or-tile state.
- Tree-layout invariant, kept in sync at map/unmap/float/move: a window is a leaf in its `workspace.root` iff `floating_geom is None`. Floats and fullscreen live outside the tree; fullscreen is a render-only override (the window keeps its leaf, and `apply_geometry` early-returns after sizing it to the full box, leaving the hidden tiles untouched until exit).
- Tileâ†’float detach is centralized in `float_client(client)` (seed `floating_geom` from the current outer rect, then `layout.remove` the leaf); called by `toggle_floating` and mouse move/resize start (`begin_dragging_client`/`begin_resizing_client`). Route new tileâ†’float entry points through it â€” setting `floating_geom` on a tiled window without removing its leaf strands a ghost leaf and breaks the leaf-iff-`floating_geom is None` invariant (a fixed bug). Its `workspace is not None` guard covers orphaned tiled windows (no tree to remove from).
- Keybindings live in `server.bindings`, a `{(modifier_mask, keycode): action}` dict built by `key_bindings` (user-extensible via `@wel.override`); `keyboard_key` looks up `(mods, keycode)` on **press only** while unlocked, else forwards the key. Edge-triggered: a held key fires once, no auto-repeat. Raw media keys carry no modifier (mods=0) and dispatch the same way. `build_keycode_map` resolves xkb **sym names** -> evdev keycodes from the default keymap (`xkb_keymap_new_from_names(ctx, NULL, 0)`, layout 0/level 0 to strip Shift); that keymap already includes XF86 media keys, so `server.keycode["XF86AudioRaiseVolume"]`/`LowerVolume`/`Mute` (evdev 113/114/115) resolve with no special handling -- verified via `xkbcli compile-keymap`, since presence depends on system xkb data, not welpy source.
- A window's current size is cached in `Client.inner_size` -- the inner/content `(width, height)` tuple (its box minus the border on each side; `None` before first map), set by `resize_client` each layout pass. Readers use it to get a window's size directly instead of re-deriving geometry via `layout.walk`: clip sizing (`apply_clip`), configure dedup (`set_size`), X11 reconfigure (`client_map`/`drag_client`), and `group_window`'s long-side test (where the symmetric border cancels in the width-vs-height comparison).

## Conventions

- Multiline parenthesized expressions put the opening `(` alone on its line, operands indented one level, and the closing `)` dedented on its own line (not trailing the last operand). **Why:** the chosen readability style here; existing code is mixed, so it isn't reliably inferable from a sample. **How to apply:** any parenthesized expression spanning lines, e.g. a conditional/ternary assignment -- the `axis = (...)` ternary in `layout.move` (and `wel.group_window`) is a worked example; note `layout.cycle_layout` predates the rule and still trails its closing paren.
- `layout.py` docstrings describe tree operations only; UI keybindings (`mod+v` etc.) are named in `wel.py` handler docstrings and `key_bindings`, never in the pure layout module. **Why:** key mentions in `wrap`/`unwrap`/`cycle_layout` silently went stale when bindings were rebound (group `mod+w`->`mod+v`, axis-flip->`mod+e`); the pure module has no reason to track UI and naming keys there only invites rot. **How to apply:** when adding or renaming a layout helper, don't cite a key in its docstring.
- Standalone planning/spec/review docs and the session transcript (`session.jsonl`) are throwaway working artifacts, removed from the repo once work lands; durable design rationale lives in code comments and `MEMORY.md`. **Why:** keeps the tree to source + tests and avoids stale design docs. **How to apply:** don't expect a committed plan/spec for a feature; delete such files (and the session log) with the landing commit.
- `pytest` and `pylint` are pinned in the project `shell.nix` (alongside `deps.nix` and the reference compositors), so they run via plain `nix-shell --run "pytest tests.py"` / `nix-shell --run "pylint ."`, never `nix-shell --packages ...`. **Why:** `--packages python3Packages.pytest` skips the project's pinned deps and can diverge from the real toolchain. **How to apply:** run either through `nix-shell --run`. The suite mocks the wlroots C bindings (no real backend/socket/subprocess), so both are read-only and non-destructive.
- Pure-Python binding helpers that take an `ffi` (e.g. `bindings.wl_list_for_each`) are unit-tested with a throwaway **ABI-mode** `cffi.FFI()` -- a plain `ffi.cdef(...)` + `ffi.new(...)`, no `set_source`/compile and no wlroots headers. **Why:** the suite's `ffi`/`lib` are `MagicMock`s that can't do real `offsetof`/`addressof`/pointer arithmetic, so the container_of logic would otherwise go unexercised (or hang on a mock `while node != head`). **How to apply:** to test an ffi-taking helper's logic, build a real FFI with a minimal struct cdef; mock only the wlroots-specific surface so higher-level callers (`client_rendered`) still stub via `patch`.

## Gotchas

- Fullscreen/resize pointer landmine: re-pointing pointer focus runs synchronously (e.g. `toggle_fullscreen` â†’ `apply_focus`) *before* the client commits its new-size buffer. The cursor is hit-tested against the old (already repositioned) buffer, so if it falls outside, focus is cleared and nothing re-samples it until the next input â€” scroll and the first click are lost. Mitigated by `rebase_pointer` at the input dispatch sites.
- cffi pointer equality: `==` compares by address and is safe against `None` (`ffi.NULL == None` is False; a non-null cdata `== None` is False). `rebase_pointer`'s dedup relies on this.
- `ffi.addressof(cdata, "field")` requires `cdata` to be a struct *value*, not a pointer; to take the address of a field reachable through a pointer `p`, dereference first -- `ffi.addressof(p[0], "field")`. `client_rendered` uses `ffi.addressof(surface[0], "current_outputs")` for exactly this. Partial structs declared with `...` still resolve `ffi.offsetof`/`addressof` (offsets come from the real headers at compile).
- `_path(root, leaf)` returns `(container, index)` pairs, yet `_collapse(path)` re-derives each container's index in its parent live via `_index` instead of reusing the carried index. **Why:** `move` inserts the relocated leaf into a path container *before* calling `_collapse`, which shifts later positions, so a stored index would drop/promote the wrong child; `remove` has no pre-collapse insert and would be safe, but `_collapse` is shared by both. Don't "optimize" `_collapse` to trust the path's indices.
- `Client` (`@dataclass(kw_only=True)`) and `layout.Container` (`@dataclass`) both default to `eq=True, frozen=False`, so instances are **unhashable** (`__hash__` is None) and `==` compares by field value. `layout.py` consequently operates on tree leaves by identity only: it never hashes a leaf (no leaf-keyed `dict`/`set`; `leaves`/`adjacent_leaves` build lists, never `dict(walk(...))`) and never leans on `==` (`_index`/`_path`/`container_of` use `is`). A future leaf-keyed dict or `list.remove(leaf)` would raise "unhashable type" or drop the wrong node. `make_client()` builds real `XdgClient` dataclasses, not mocks, so tests hit the same constraint.

## Decisions

- Stale-pointer-after-fullscreen fix is a dispatch-site rebase (`rebase_pointer`) in `cursor_button` (before `notify_button`) and `cursor_axis` (before `notify_axis`, skipped while a window is grabbed), not a commit-time rebase. Rationale: dispatch-site covers X11 too (no `pending_serial` dependency), fixes both symptoms, and dedups (no-op when focus already matches) to avoid spurious motion events. Accepted trade-off: hover / cursor-image isn't refreshed until the next click or scroll.
- Cross-compositor reference for input-focus work: dwl re-points pointer focus synchronously (e.g. in `arrange()`/`focusclient()` via `motionnotify(0,...)`), never on the resize-completing commit, so it shares the same latent fullscreen ordering bug; sway rebases (`cursor_rebase_all()`) after a transaction applies, once all resized buffers land.
- Tree layout uses an n-ary container tree (sway/i3 model). Binary BSP was rejected because tabbed/stacked containers (wanted "maybe later") are inherently n-ary and binary needs rebalancing for equal N-way splits; stateless spiral/fibonacci layouts were rejected as not interactively manipulable.
- `ContainerLayout` is orientation-only (`HORIZONTAL`/`VERTICAL`); tabbed/stacked were deliberately kept out of the enum as a separate future concept, not v1.
- Grouping is two keys (replacing the earlier `mod+w` create/destroy toggle): `mod+v` wraps the focused window alone in a new container split along its long side, a **no-op when the window has no siblings** (so it never pointlessly nests a lone window and can't double-wrap -- after one wrap the window is alone in its new group); `mod+e` flips a container's axis. There is deliberately **no explicit ungroup** -- groups dissolve only when collapse empties them or drops them to one child. `layout.unwrap` consequently has no production caller (the retired toggle's destroy branch was its only one) and survives only as the tested `wrap`/`unwrap` pair.
- Directional focus (`mod+hjkl`) is **structural**, sharing `move`'s `_neighbor` climb: it ascends to the nearest ancestor that splits on the move axis with a sibling that way, then focuses that sibling's **most-recently-focused** window (max `focus_order`). `layout.adjacent_leaves` returns the sibling's windows and `focus_direction` applies the `focus_order` policy, keeping `layout` free of `Client`. No-op at an edge -- unlike `move`, focus has no parent-escape (nothing to land on). This **reverses** the earlier geometric center-to-center "nearest leaf beyond the focused center" rule, now removed along with its `_center`/`_beyond` helpers (i3's overlap tie-break and cross-monitor edge-jump were already dropped). Rationale for the flip: symmetry with `move`, a "return to where you were" landing, and geometry can't disambiguate co-located leaves once tabbed/stacked containers land (they share one rect, so `walk` gives identical centers).
- Tree traversal is search-on-demand (DFS from the workspace root) rather than parent back-pointers: trees are tiny (one workspace's windows), so O(n) ascent is cheap and there are no back-refs to keep in sync.
- Tree-layout collapse is the shared `_collapse(path)` helper, run by both `remove` and `move`: it tidies **only the affected leaf's ancestor chain** (drop emptied containers, promote single-child ones), never a global single-child flatten, and never the root. Rationale: a global pass would dissolve an unrelated one-window `mod+v` group on any close; localized leaves it intact. Empty/1-child cleanup is mandatory regardless because a `mod+v` wrap can leave a 1-child container that later empties.
- Directional move (`mod+shift+hjkl`, implemented: pure `layout.move(root, leaf, direction)` + the `move_direction` handler) is **structural relocate**, sway-style -- chosen over geometric swap (can't pop a window into open space or restructure nesting) and i3-style reorder-only (can't descend into a sibling container). Model: climb from the window's parent to the first ancestor whose split axis matches the move *and* has a neighbor on the travel side, then reorder past a leaf neighbor, descend into a container neighbor, or pop the window out beside its subtree, then `_collapse` the drained source chain. Locked rules: descend landing is purely structural, no geometry -- perpendicular sibling -> index 0, matching-axis sibling -> facing end (D1); at the edge of every matching-axis ancestor the leaf escapes its immediate parent one level up toward the move side, and only a leaf already directly under the root no-ops (no implicit root split) (D2); only the focused leaf moves, collapse tidies the source (D3). Subtlety: for descend/pop-out, detach the leaf *before* placing it so the receiving ancestor never momentarily looks single/empty and survives `_collapse`.

- `client_holds_paint` gates the screen-hold on `client_rendered` (surface present on â‰Ą1 output, i.e. non-empty `wl_surface.current_outputs`, walked by `bindings.wl_list_for_each`) plus a `!= Layer.FLOAT` opt-out -- replacing the earlier coarse `Layer.TILE and workspace.fullscreen is None` proxy. This makes **fullscreen entry atomic** (the entering window stays on the output through the transition, so it holds until its full-size buffer lands -- one frame, no flash showing the old size) and is provably freeze-free: wlroots gates both `wlr_scene_buffer_send_frame_done` and the output-enter/`current_outputs` tracking on the same occlusion-aware `node.visible` region, so a window holds **iff** it's visible **iff** it gets frame-done **iff** it will ack -- an occluded peer (the b766954 freeze) can't hold by construction, regardless of opaque regions. The `!= Layer.FLOAT` clause preserves the deliberate "responsive interactive resize" choice (a float being dragged must not stall the screen). Chosen over (B) merely special-casing the fullscreen window inside the coarse proxy, and (C) a sway-style deferred per-window transaction applying scene changes only on configure-ack: A retires the proxy wholesale, is correct for every layer, and keeps welpy's hold-the-whole-screen model. `wl_list_for_each(ffi, head, ctype, member)` is a generic intrusive-`wl_list` iterator in `bindings.py` (logic-in-Python per the binding convention; no bespoke C helper); `client_rendered` is its first caller (`any(...)` for non-empty). A bare `head.next != head` emptiness test was the rejected minimal alternative -- the generic iterator was chosen for reuse, so `client_rendered`'s `any(...)` is deliberate; don't "optimize" it back to an inline emptiness check (the path is rarely hit anyway, gated behind an unacked resize). The hold predicate now reads scene state, not just workspace bookkeeping, so its unit tests stub `wel.client_rendered`.
