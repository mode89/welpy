## Architecture

- The `welpy/` modules layer as an acyclic DAG that stays acyclic with no callback injection between modules because the shared client-lookup queries (`clients_in`/`clients_visible`/`client_monitor`) live in `model` -- so `geometry` is a clean `model`+`layout` leaf -- and `focus` isn't the hub it appears (`focus.reconcile`'s `activate_lock`/`activate_unmanaged` read only `Server` fields). `key_bindings`/`modkey` sit in `app` at the top because they reference `commands` + lifecycle (placing them lower would cycle), so `input` reads the binding table off `server.bindings` and `commands` is driven by `app`, neither importing the other.
- Cross-module deps the linear DAG order hides: `focus` also imports `layout`+`ext_workspace` (`recent_tiled_leaf`/`active_container` use `layout`; `focus.reconcile` calls `ext_workspace.publish`), and `output` also imports `session_lock`+`ext_workspace` (`output.reconcile` drives lock layout + workspace publish).
- Intra-package imports are relative (`from . import layout`); `welpy/__init__.py` defines the `override`/`_install` hook.
- `tests/__init__.py` exists so `from tests.helpers import ...` resolves under pytest's prepend import mode.
- `welpy/layout.py` and `welpy/bindings.py` have no mirror test file; they're exercised within `tests/test_app.py`.
- Reference implementations consulted for behavior parity: dwl, sway, qtile. Source via nix: `nix-build '<nixpkgs>' -A dwl.src` (sway: `-A sway-unwrapped.src`; qtile: `-A python3Packages.qtile.src`).
- Pointer focus pipeline: `forward_pointer_motion(server, time)` re-points the seat pointer to the surface under the cursor; called from the pointer-motion handlers (`cursor_motion`, `cursor_motion_absolute`, and the `cursor_button` release path) and from `focus.reconcile` (re-sample after the scene changes with no mouse move). `surface_at` resolves the surface via `wlr_scene_node_at`.
- `wlr_pointer_constraints_v1_constraint_for_surface(constraints, surface, seat)` resolves the pointer constraint bound to a surface, so the active constraint is looked up from the pointer-focused surface rather than by walking `pointer_constraints->constraints` (dwl walks the list manually; sway uses this helper).
- Pointer motion funnels through one core, `process_pointer_motion(server, device, delta, unaccel, time_msec)`: `cursor_motion` (relative) and `cursor_motion_absolute` (absolute -> layout delta) are thin shims over it that stream the raw delta to relative-pointer clients, enforce the active pointer constraint, then move the cursor + forward the hover (or `drag_client` while grabbed).
- `apply_pointer_constraint(server, dx, dy)` is the pointer-lock override seam: returns the possibly region-confined delta, or `None` when the pointer is locked (caller pins the cursor and skips the focus re-point).
- `server.active_constraint` is `None` or the raw `wlr_pointer_constraint_v1` currently in effect (equal to the `.constraint` of one of the `PointerConstraint` records in `server.constraints`), not a record itself. Invariant: when `server.constraints` is empty, `active_constraint` is `None` -- a record and its active flag clear together in `constraint_destroy` -- so `apply_pointer_constraint` early-returns on empty `server.constraints`, skipping the per-motion `constraint_for_surface` cffi call in the no-lock common case without dropping a pending deactivation.
- Scene hit-testing (`wlr_scene_node_at`) uses the committed surface size clamped by the clip; welpy always sets a clip (`wlr_scene_subsurface_tree_set_clip`), and the clip only *shrinks* the hit region, never grows it. So a window's hittable area lags its configured size until the client commits the new buffer.
- `bump_focus_order` only bumps `client.focus_order`; all focus effects (keyboard focus, raise, border colors, and the pointer re-point) are emitted by `focus.reconcile` at the handler boundary.
- Outside an interactive grab (during a move/resize grab, motion drives `drag_client` instead), welpy re-points pointer focus on every motion — no implicit-grab / dwl-style "CurPressed" focus lock — so pointer focus is not held on a surface across a button press.
- `XdgClient.pending_serial` tracks a compositor-driven configure until the client renders it (the per-client analog of a transaction). Serial tracking and the screen-hold (`client_holds_paint`) are xdg-only; `X11Client` has no configure-serial tracking.
- Window tiling is a per-workspace n-ary tree: `Workspace.root` is a `layout.Container`, and `geometry.reconcile` walks it over `window_area` with equal splits along each container's axis (HORIZONTAL/VERTICAL). It replaced the retired stateless `master_stack`/`zoom`/`cycle_focus`, all now gone.
- `layout.py` holds the pure tree logic: leaves are opaque (a `Container` is told from a leaf by `isinstance` only, never by reading leaf fields, so `layout.py` never imports `Client`), and the handler modules (`commands`/`geometry`/`windows`) are the side-effecting boundary that run the helpers on each `Workspace.root`.
- `server.clients` is a flat registry only; window z-order, focus, and focus-on-close are driven by `focus_order`, a monotonic counter bumped on each focus (higher = more recent).
- `workspace.fullscreen` and per-window `floating_geom` are orthogonal: `client_layer` resolves a window as fullscreen → float → tile, and fullscreen overlays geometry without clearing `floating_geom`, so leaving fullscreen restores the prior float-or-tile state.
- Tree-layout invariant, kept in sync at map/unmap/float/move: a window is a leaf in its `workspace.root` iff `floating_geom is None`. Floats and fullscreen live outside the tree; fullscreen is a render-only override (the window keeps its leaf, and `geometry.reconcile` early-returns after sizing it to the full box, leaving the hidden tiles untouched until exit).
- Tile→float detach is centralized in `float_client(client)` (seed `floating_geom` from the current outer rect, then `layout.remove` the leaf); called by `toggle_floating` and mouse move/resize start (`begin_dragging_client`/`begin_resizing_client`). Route new tile→float entry points through it — setting `floating_geom` on a tiled window without removing its leaf strands a ghost leaf and breaks the leaf-iff-`floating_geom is None` invariant (a fixed bug). Its `workspace is not None` guard covers orphaned tiled windows (no tree to remove from).
- `server.bindings` is a flat `{(modifier_mask, code): action(server)}` dict built by `key_bindings` (user-extensible via `welpy.override`); keyboard keycodes and mouse-button codes share the one dict.
- Both `keyboard_key` (press only, while unlocked) and `cursor_button` (mouse buttons) resolve a press through `lookup_binding(server, mods, code)` -- the action-or-None seam (None forwards the input to the focused app), defined below both callers per the "callees below callers" order. It is the documented override point for modal submaps.
- Keybinding dispatch is edge-triggered -- a held key fires once, no auto-repeat -- and raw media keys carry no modifier (mods=0), dispatching the same way.
- `server.passthrough` (toggled by `toggle_passthrough`, bound to mod+shift+p) makes `lookup_binding` return None for every action except `toggle_passthrough` itself, forwarding all input to the focused app; the still-live toggle is the escape hatch. Built for nested/dev sessions.
- `build_keycode_map` resolves xkb **sym names** -> evdev keycodes once (from the default keymap `xkb_keymap_new_from_names(ctx, NULL, 0)`, layout 0/level 0 to strip Shift) so bindings can key off names. That keymap already includes XF86 media keys, so `server.keycode["XF86AudioMute"]`/`LowerVolume`/`RaiseVolume` (evdev 113/114/115) resolve with no special handling -- their presence depends on system xkb data, not welpy source (verified via `xkbcli compile-keymap`).
- A window's current size is cached in `Client.inner_size` -- the inner/content `(width, height)` tuple (its box minus the border on each side; `None` before first map), set by `resize_client` each layout pass. Readers use it to get a window's size directly instead of re-deriving geometry via `layout.walk`: clip sizing (`apply_clip`), configure dedup (`set_size`), X11 reconfigure (`xwayland.on_request_configure`/`drag_client`), and `group_window`'s long-side test (where the symmetric border cancels in the width-vs-height comparison).
- `bindings.wl_list_for_each(ffi, head, ctype, member)` is a generic intrusive `wl_list` iterator (the libwayland container_of idiom), logic-in-Python with no bespoke C helper. `client_rendered` is its only caller, walking a surface's `current_outputs` with `any(...)` to test on-screen presence.

## Conventions

- Multiline parenthesized expressions put the opening `(` alone on its line, operands indented one level, and the closing `)` dedented on its own line (not trailing the last operand). **Why:** the chosen readability style here; existing code is mixed, so it isn't reliably inferable from a sample. **How to apply:** any parenthesized expression spanning lines, e.g. a conditional/ternary assignment -- the `axis = (...)` ternary in `layout.move` (and `commands.group_window`) is a worked example; note `layout.cycle` predates the rule and still trails its closing paren.
- `layout.py` docstrings describe tree operations only; UI keybindings (`mod+v` etc.) are named in command/handler docstrings and `app.key_bindings`, never in the pure layout module. **Why:** key mentions in `wrap`/`unwrap`/`cycle` silently went stale when bindings were rebound (group `mod+w`->`mod+v`, axis-flip->`mod+e`); the pure module has no reason to track UI and naming keys there only invites rot. **How to apply:** when adding or renaming a layout helper, don't cite a key in its docstring.
- Standalone planning/spec/review docs and the session transcript (`session.jsonl`) are throwaway working artifacts, removed from the repo once work lands; durable design rationale lives in code comments and `MEMORY.md`. **Why:** keeps the tree to source + tests and avoids stale design docs. **How to apply:** don't expect a committed plan/spec for a feature; delete such files (and the session log) with the landing commit.
- `pytest` and `pylint` are pinned in the project `shell.nix` (alongside `deps.nix`), so they run via plain `nix-shell --run "pytest"` / `nix-shell --run "pylint ."`, never `nix-shell --packages ...`. **Why:** `--packages python3Packages.pytest` skips the project's pinned deps and can diverge from the real toolchain. **How to apply:** run either through `nix-shell --run`. The suite mocks the wlroots C bindings (no real backend/socket/subprocess), so both are read-only and non-destructive.
- pylint `duplicate-code` (R0801) on shared test-setup boilerplate across the per-module test files is suppressed with an inline `# pylint: disable=duplicate-code` at each flagged spot, never a global `.pylintrc` disable. **Why:** a global disable also blinds genuine duplication in `welpy/` source, and per-file/path scoping is unreliable for R0801 -- pylint attributes a cross-file cluster to a single (sometimes a third) file, and a module-level disable placed wrong (e.g. inside the module docstring) silently no-ops. **How to apply:** when duplicated fixture/setup blocks trip R0801, disable it inline at the test functions involved.
- Pure-Python binding helpers that take an `ffi` (e.g. `bindings.wl_list_for_each`) are unit-tested with a throwaway **ABI-mode** `cffi.FFI()` -- a plain `ffi.cdef(...)` + `ffi.new(...)`, no `set_source`/compile and no wlroots headers. **Why:** the suite's `ffi`/`lib` are `MagicMock`s that can't do real `offsetof`/`addressof`/pointer arithmetic, so the container_of logic would otherwise go unexercised (or hang on a mock `while node != head`). **How to apply:** to test an ffi-taking helper's logic, build a real FFI with a minimal struct cdef; mock only the wlroots-specific surface so higher-level callers (`client_rendered`) still stub via `patch`.
- To assert cffi **out-param** math under the mock `lib`/`ffi` harness (e.g. confine/cursor-hint deltas), set `server.ffi.new.side_effect = real.new` from a throwaway `cffi.FFI()` so `ffi.new("double *")[0]` round-trips, then give the C-helper mock a `side_effect` that writes concrete values into the out-pointers. **Why:** MagicMock cdata doesn't remember `[0]` writes, so the delta arithmetic would otherwise be mock-on-mock and untestable. **How to apply:** only `ffi.new` needs to be real; `ffi.cast`/`addressof`/`NULL` stay mocked.
- Tests patch by the canonical dotted string (`patch("welpy.app.X")` / `patch("welpy.<mod>.X")`) and reference symbols by their real home (`app.setup`, `model.Layer`, `layout.Rect`). **Why:** a `patch(...)` target *string* is a module path, not an attribute ref, so it must name the symbol's owning module. **How to apply:** patch the real dotted path; import each symbol from its owning module.
- `welpy.override` `setattr`s the hook's home module, so a cross-module qualified call (`focus.reconcile`) or a same-module bare call sees the override, but `from .focus import reconcile` elsewhere silently binds the pre-override original (no error). Types are imported by-name from their home (`from .model import Server`); tests do the same (`model.Layer`, `layout.Rect`). The qualify-customization-points rule (incl. tunable constants) is in `AGENTS.md`.
- welpy's leaf modules are dependency **sinks**: `layout`/`ext_workspace`/`libinput` never import the compositor; `ext_workspace` inverts the edge via injected callbacks (`on_activate`/`on_assign`) rather than calling back in. **Why:** keeps the import graph acyclic (a pylint `cyclic-import` gate). **How to apply:** a new module needing core behavior takes a callback or stays a pure leaf, never importing `app`.

## Gotchas

- Fullscreen/resize pointer landmine: re-pointing pointer focus runs synchronously (e.g. `toggle_fullscreen` → `focus.reconcile`) *before* the client commits its new-size buffer. The cursor is hit-tested against the old (already repositioned) buffer, so if it falls outside, focus is cleared and nothing re-samples it until the next input — scroll and the first click are lost. Mitigated by `rebase_pointer` at the input dispatch sites.
- cffi pointer equality: `==` compares by address and is safe against `None` (`ffi.NULL == None` is False; a non-null cdata `== None` is False). `rebase_pointer`'s dedup relies on this.
- `ffi.addressof(cdata, "field")` requires `cdata` to be a struct *value*, not a pointer; to take the address of a field reachable through a pointer `p`, dereference first -- `ffi.addressof(p[0], "field")`. `client_rendered` uses `ffi.addressof(surface[0], "current_outputs")` for exactly this. Partial structs declared with `...` still resolve `ffi.offsetof`/`addressof` (offsets come from the real headers at compile).
- Compiling against `wlr_pointer_constraints_v1.h` needs the generated `<wayland-protocols/pointer-constraints-unstable-v1-enum.h>`, which wlroots does **not** install -- generate it with `builder.enum_header("wayland-protocols", "unstable/pointer-constraints/pointer-constraints-unstable-v1.xml", "pointer-constraints-unstable-v1")`, the xdg-shell pattern. The relative-pointer header includes only `wayland-server-core.h`, so it needs none.
- `wlr_pointer_constraint_v1_send_deactivated()` may synchronously destroy the constraint (oneshot lifetime), firing its `destroy` listener mid-call -- a constraint activate/deactivate transition must tolerate its own active reference being cleared during the deactivate.
- `_path(root, leaf)` returns `(container, index)` pairs, yet `_collapse(path)` re-derives each container's index in its parent live via `_index` instead of reusing the carried index. **Why:** `move` inserts the relocated leaf into a path container *before* calling `_collapse`, which shifts later positions, so a stored index would drop/promote the wrong child; `remove` has no pre-collapse insert and would be safe, but `_collapse` is shared by both. Don't "optimize" `_collapse` to trust the path's indices.
- `layout.move`'s descend/pop-out path detaches the leaf from its source parent *before* inserting it under the receiving ancestor, so that ancestor is never momentarily single-child/empty and survives the trailing `_collapse`. Inserting before detaching would let `_collapse` drop or promote the wrong node.
- `Client` (`@dataclass(kw_only=True)`) and `layout.Container` (`@dataclass`) both default to `eq=True, frozen=False`, so instances are **unhashable** (`__hash__` is None) and `==` compares by field value. `layout.py` consequently operates on tree leaves by identity only: it never hashes a leaf (no leaf-keyed `dict`/`set`; `leaves`/`adjacent_leaves` build lists, never `dict(walk(...))`) and never leans on `==` (`_index`/`_path`/`container_of` use `is`). A future leaf-keyed dict or `list.remove(leaf)` would raise "unhashable type" or drop the wrong node. `make_client()` builds real `XdgClient` dataclasses, not mocks, so tests hit the same constraint.
- Passthrough (and any future submap) exempts its own toggle by **identity** (`action is not toggle_passthrough`), so that binding must store the bare function, not a lambda wrapping it -- a lambda breaks the exemption and traps input.
- Adding a built-in keybinding that references a new key name fails ~23 `setup`/`make_server` tests with `KeyError` until the name is added to `make_keycode_map` in `tests/helpers.py` (it lists only keys used by built-in bindings); a new `Server` field likewise needs a default added to `make_server` (also in `tests/helpers.py`).
- pylint's `line-too-long` gate is 80 cols (`.pylintrc`), but it exempts any physical line carrying a trailing `# pylint:` pragma, so some 100+ char lines stay unflagged -- an over-long line isn't necessarily a new regression.
- `geometry.configure_x11` carries no leading underscore despite being a low-level helper, because `xwayland`/`input` call it cross-module and a leading `_` trips pylint `protected-access` (W0212) at the caller.
- A module-level `from welpy import <mod>` / `from . import <mod>` shadows any local of the same name -- `patch(...) as <mod>:` mock vars, plain assignments, or a `setup()` C-server local. pylint flags W0621, and code that also calls the real `<mod>.fn(...)` hits `UnboundLocalError`; the convention is to rename the local (mock vars to the patched symbol e.g. `as focus_client`, source C-server locals to `<mod>_server`), leaving the module's `<mod>.X` calls intact.
- A `welpy/` submodule named after a Python builtin -- `welpy/input.py` -- makes `from . import input` (and `from welpy import (... input)`) trip pylint W0622 redefined-builtin; every such import line carries a `# pylint: disable=redefined-builtin` pragma (in `app.py`, `tests/test_input.py`, and `tests/test_app.py`). The builtin is unused in this codebase, so the shadow is harmless.
- The qualify-customization-points rule (read `model.X` by qualified access so configs can rebind) applies only to **tunable** constants (`BORDER_*`, `WORKSPACE_NAMES`, scales); **structural** constants like `SHELL_LAYERS` (the fixed layer-order tuple) aren't customization points, so they needn't be read qualified for rebind (modules read `SHELL_LAYERS` by-name or qualified interchangeably).

## Decisions

- Stale-pointer-after-fullscreen fix is a dispatch-site rebase (`rebase_pointer`) in `cursor_button` (before `notify_button`) and `cursor_axis` (before `notify_axis`, skipped while a window is grabbed), not a commit-time rebase. Rationale: dispatch-site covers X11 too (no `pending_serial` dependency), fixes both symptoms, and dedups (no-op when focus already matches) to avoid spurious motion events. Accepted trade-off: hover / cursor-image isn't refreshed until the next click or scroll.
- Cross-compositor reference for input-focus work: dwl re-points pointer focus synchronously (e.g. in `arrange()`/`focusclient()` via `motionnotify(0,...)`), never on the resize-completing commit, so it shares the same latent fullscreen ordering bug; sway rebases (`cursor_rebase_all()`) after a transaction applies, once all resized buffers land.
- Pointer-lock (`wlr_pointer_constraints_v1` + `wlr_relative_pointer_v1`) unifies `cursor_motion`/`cursor_motion_absolute` into one `process_pointer_motion` core (dwl `motionnotify` shape). Rejected: surgical per-handler insertion (keeps the two handlers' duplication) and a lock-only MVP.
- Pointer-lock scope is full (relative + locked + confined); confinement uses `wlr_region_confine`, available because `pixman-1` is already in bindings `_PKGS`.
- Pointer-lock constraint tracking follows dwl/sway: a destroy listener per constraint registered at `new_constraint`, plus one `active_constraint` resolved by pointer focus. An "active-only" single listener was rejected because `send_deactivated` can destroy a constraint and inactive constraints still need their own destroy cleanup.
- Pointer-lock ships no unlock keybinding: keybindings keep firing while the pointer is locked (`keyboard_key` dispatches bindings before forwarding) and a constraint is active only while its surface holds pointer focus, so any keyboard-driven focus change (workspace switch, close, focus-other) deactivates it -- that focus-tied deactivation is the escape, matching dwl/sway (neither ships one). Reversed an initial instinct to add an explicit escape hatch.
- Pointer-lock honors the client cursor-hint: on destroy of the *active* constraint (not on focus-loss deactivation), warp the cursor to the hint so it reappears where the app expects (dwl `cursorwarptohint`); surface-local hint -> layout via the client's `content_tree.node` origin (approximate for non-trivial xdg geometry offsets, acceptable for a hint).
- Pointer-lock fully unifies the absolute path: `cursor_motion_absolute` converts to a layout delta + `wlr_cursor_move` (not `wlr_cursor_warp_absolute`) so a lock holds regardless of input source and nested sessions carry constraints (dwl's model). Reactive-only fallback if touch/tablet mapping regresses: keep `warp_absolute` and route only relative motion through constraints.
- Tree layout uses an n-ary container tree (sway/i3 model). Binary BSP was rejected because tabbed/stacked containers (wanted "maybe later") are inherently n-ary and binary needs rebalancing for equal N-way splits; stateless spiral/fibonacci layouts were rejected as not interactively manipulable.
- `ContainerLayout` is orientation-only (`HORIZONTAL`/`VERTICAL`); tabbed/stacked were deliberately kept out of the enum as a separate future concept, not v1.
- Grouping is two keys (replacing the earlier `mod+w` create/destroy toggle): `mod+v` (`group_window`) wraps the focused window alone in a new container split along its long side -- a **no-op when the window has no siblings**, so it never pointlessly nests a lone window and can't double-wrap; `mod+e` (`cycle_layout`) flips a container's axis. There is deliberately **no explicit ungroup** -- groups dissolve only when collapse empties a container or drops it to one child.
- `layout.unwrap` has no production caller: the retired `mod+w` toggle's destroy branch was its only one, and the no-explicit-ungroup decision left nothing to replace it. It survives only as the tested `wrap`/`unwrap` pair.
- Directional focus (`mod+hjkl` -> `focus_direction`) is **structural**, sharing `move`'s `_neighbor` climb: it ascends to the nearest ancestor that splits on the move axis with a sibling that way, then focuses that sibling's **most-recently-focused** window (max `focus_order`). `layout.adjacent_leaves` returns the sibling's windows and `focus_direction` applies the `focus_order` policy, keeping `layout` free of `Client`. No-op at an edge -- unlike `move`, focus has no parent-escape, nothing to land on. Rationale: symmetry with `move` and a "return to where you were" landing.
- Focus-on-close lands on the **lineage-local MRU**: closing a tiled window focuses (via `layout.successor`) the most-recently-focused window in the nearest enclosing container that still holds another -- not the workspace-global MRU -- to keep focus where the user was working. A closed float, or a tile whose tree empties, falls back to `top_client`. Rejected: pure global MRU (jumps to a recently-touched window elsewhere) and a per-container focus stack (deferred until tabbed/stacked exist).
- `layout.successor(root, leaf, select)` both climbs the lineage and returns the single max-by-`select` window, deliberately unlike the `adjacent_leaves`/`focus_direction` split (layout returns candidates, the caller applies the `focus_order` policy): the close path wants one window, and passing `select` (`lambda c: c.focus_order`) keeps `layout` free of `Client`. Intentional -- don't "restore" a candidate-list shape.
- Tree traversal is search-on-demand (DFS from the workspace root) rather than parent back-pointers: trees are tiny (one workspace's windows), so O(n) ascent is cheap and there are no back-refs to keep in sync.
- Tree-layout collapse is the shared `_collapse(path)` helper, run by both `remove` and `move`: it tidies **only the affected leaf's ancestor chain** (drop emptied containers, promote single-child ones), never a global single-child flatten, and never the root. Rationale: a global pass would dissolve an unrelated one-window `mod+v` group on any close; localized leaves it intact. Empty/1-child cleanup is mandatory regardless because a `mod+v` wrap can leave a 1-child container that later empties.
- Directional move (`mod+shift+hjkl` -> `move_direction` -> pure `layout.move`) is a **structural relocate**, sway-style -- chosen over geometric swap (can't pop into open space or restructure nesting) and i3 reorder-only (can't descend into a sibling container). It climbs from the leaf's parent to the first ancestor whose split axis matches the move with a neighbor on the travel side, then reorders past a leaf neighbor, descends into a container neighbor (perpendicular -> index 0, matching-axis -> facing end), or -- at the edge of every matching-axis ancestor -- pops the leaf one level up toward the move side; then `_collapse` tidies the drained source. Only the focused leaf moves; landing is purely structural (no geometry). The sole no-op is a leaf directly under the root with no same-axis sibling on the travel side -- nothing to reorder past, and nothing above the root to escape into (no implicit root split); a root child *with* a same-axis neighbor still reorders.
- `client_holds_paint`'s screen-hold predicate is `client_rendered` (surface on >=1 output) AND `!= Layer.FLOAT`, replacing the coarse `Layer.TILE and workspace.fullscreen is None` proxy. **Why:** fullscreen entry becomes atomic -- the window holds the screen through the transition until its full-size buffer lands, no one-frame flash of the old size. Provably freeze-free: wlroots gates `wlr_scene_buffer_send_frame_done` and `current_outputs` tracking on the same occlusion-aware `node.visible` region, so a window holds iff it's visible iff it will ack -- an occluded peer (the b766954 freeze) can't hold by construction. The `!= Layer.FLOAT` opt-out keeps interactive resize of a dragged float responsive. Rejected: special-casing only the fullscreen window in the coarse proxy; a sway-style deferred per-window transaction on configure-ack. Tests stub `output.client_rendered` since the predicate reads scene state.
- The binding seam is action-level (`lookup_binding(server, mods, code) -> action | None`), chosen over a table-returning `active_bindings(server) -> Mapping`: config can then express "swallow unbound keys in a submap" as a one-line inline default (`table.get(key, lambda s: None)`), where a table hook would need a custom mapping class.
- Modal submaps (sway-style modes / prefix keys) are deliberately kept out of core -- only the `lookup_binding` seam is provided, and the mechanism (state, enter/leave, swallow, escape) lives in user config via a `welpy.override` of `lookup_binding`. Passthrough is the one built-in built on the seam.
- Passthrough is full: because `lookup_binding` is shared with `cursor_button`, it suppresses mouse-button bindings (mod+drag/resize) too, not just keys -- an accepted consequence of the shared seam, not keyboard-only.
- Intended (unbuilt) submap recipe: `enter_submap(server, table, exit_code)` with a **required** exit_code that injects `(0, exit_code) -> leave_submap` so a submap can't trap the keyboard; sticky by default (an action opts into one-shot by calling leave itself); submap keys bound mod-less `(0, code)` with the user releasing the entry modifier; unbound keys swallowed. Key-repeat (needs a timer) and clear-on-lock were deferred.
- `welpy.override` is target-only (`@welpy.override(module.hook)`): it resolves `target.__module__` to reach the hook's real home and chains by currying the previous version + rewriting `wrapper.__module__`. **Why target rather than by-name:** a by-name form silently no-ops when a hook's home module differs from where a config names it.
- The compositor launches via `python -m welpy` (`welpy/__main__.py` -> `welpy.app.main`); the package's canonical identity lets a user config import `welpy` and its submodules directly, with no `sys.modules` aliasing.

## Dead Ends

- ✗ Commit-time rebase (abandoned): re-point the pointer in `windows.on_commit` when the `pending_serial` catch-up fires (sway's transaction-apply model). Dropped — xdg-only (misses X11 fullscreen) and a second mechanism overlapping the dispatch-site rebase.
- ✗ Bare `head.next != head` emptiness test for `client_rendered` (rejected): the generic `bindings.wl_list_for_each` iterator was chosen instead for reuse, so `client_rendered`'s `any(...)` over `current_outputs` is deliberate -- don't "optimize" it back to an inline emptiness check. The path is rarely hit anyway, gated behind an unacked resize.
- ✗ Geometric directional focus (reversed in favor of the structural `_neighbor` climb): the earlier center-to-center "nearest leaf beyond the focused center" rule and its `_center`/`_beyond` helpers were removed; i3's overlap tie-break and cross-monitor edge-jump had already been dropped before that. Geometry can't disambiguate co-located leaves once tabbed/stacked containers share one rect (`walk` gives identical centers).
- ✗ A dedicated `welpy/config.py` for tunable constants (rejected): each constant lives in its owning module, read qualified so configs can rebind it; a separate config module would only add load-ordering complexity (a config runs at startup before the compositor is built) for no gain.
